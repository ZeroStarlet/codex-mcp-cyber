# 审查清单（PROMPT SSOT）

本文件是 **唯一** 完整 PROMPT 模板与五项检查的来源。`SKILL.md`、`codex-guide.md`、`examples.md` 只引用此处，不复述全文。

## 发送前校验

- 所有 `[方括号占位符]` 必须替换为实际内容再发送。
- 切勿把字面占位符（如 `[文件列表]`、`[粘贴 git diff --no-color 输出]`）发给 Codex——会导致空审或误报 PASS。
- Codex 工具**无内置审查 system prompt**，清单不可省略。

## 调用前

```bash
git diff --no-color
```

把 diff 与「改动文件 / 改动目的 / 自测证据 / 本次重点」嵌入 PROMPT。`cd` 传裸路径，不要字面引号。

**大 diff 走文件**：diff 超过约 2 万 token（千行级）时不要内嵌——写入仓库内临时文件
（如 `.scratch/review-<日期>.diff`，勿提交），PROMPT 里给出相对路径，并要求 Codex
读取该文件后与实时 `git diff --no-color` 交叉核对一致性（Codex 在工作目录内只读可用）。
复审时更新同一文件。

Windows：`os error 123` 的元凶是给 `cd` 包了**字面引号**——传裸路径即可；中文 / 非 ASCII 路径同样直接传裸路径，无需特殊处理。

## 标准 PROMPT 模板

````
请 review 以下代码改动（只审不改）：

**改动文件**：[文件列表]
**改动目的**：[简要描述]
**自测证据**：[测试 / 冒烟命令与结果摘要；无法运行则写明原因]
**本次重点**：[如：鉴权边界 / 并发竞态 / 输入校验 / 是否引入回归 —— 没有可省略]

**Git Diff**:
```diff
[粘贴 git diff --no-color 输出]
```

**请检查**：
1. 逻辑正确性
2. 边界条件（空值 / 越界 / 非法输入 / 溢出）
3. 安全风险（注入 / 越权 / 凭证泄漏 / 敏感信息泄露）
4. 测试缺口（缺少断言 / 未覆盖边界 / 回归缺失）
5. 可维护性（命名 / 结构 / 重复代码 / 过度抽象）

**请给出明确结论**：
- ✅ PASS：代码质量良好，可以合入
- ⚠️ OPTIMIZE：有优化建议但不阻塞合入，由 Claude 评估是否采纳
- ❌ CHANGE：必须修改，以下为具体问题清单
````

## 复审 PROMPT

结构与上相同。标题可改为「请复审以下修复（只审不改）：」。携带**逐条处置表**
（上轮每条意见 → 已修 / 反驳 + 证据 / 推迟 + 原因）与修复增量 diff；可引用 Codex
上一轮自己给出的标准（如「以仓库实际写法为复杂度上限」）帮助收敛。若有分歧，在
「本次重点」下增加 **与初审的分歧** 段（意见原文 + 证据引用），见
[scenarios.md](scenarios.md) 场景 E。

## 工具参数要点

| 参数 | 要求 |
|------|------|
| PROMPT | 必须含 diff + 上表完整清单 |
| cd | 裸路径 |
| sandbox | **必须** `read-only` |
| SESSION_ID | 初审 `""`；复审 = 上一轮返回值 |

完整参数与返回值见 [codex-guide.md](codex-guide.md)。
