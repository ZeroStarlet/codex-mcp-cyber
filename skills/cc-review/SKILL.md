# Codex 终审协作流程

## 定位

**Claude + Codex 双模型协作**。追求流程简单、上下文稳定、责任清晰、代码质量可控。

- **主写 = Claude（本 agent）**：需求分析、任务拆解、代码实现与修改、自测、按审查意见修复。
- **终审 = Codex**：独立 Code Review，**只审不改**。

> **术语**：Codex 的角色统称为"终审"（独立审查方）。"复审"指初审之后的任意重审轮次（第 2 轮、第 3 轮）。两者不互斥——每一轮复审都是 Codex 在履行终审职责。3 轮上限中的"终审"即第 3 次复审，规则与其他复审相同（复用 SESSION_ID、完整清单）。

## 核心准则：Karpathy 工程准则（每次必守）

写码 / 修复 / 重构前先过一遍这套准则。Codex 端也按同一标准审查——两边对齐。

完整准则见 [karpathy-guidelines.md](karpathy-guidelines.md)。摘要：

1. **想清再写**：显式说明假设；有歧义先提出别默默选；有更简方案就直说。
2. **简单优先**：只写解决问题的最小代码；不加未要求的特性 / 抽象 / 配置 / 不可能分支的错误处理。
3. **外科手术式改动**：每行改动都能追溯到需求；不顺手重构没坏的代码、不改无关格式；只清理自己产生的孤儿，不动既有死代码（除非被要求）。
4. **目标驱动**：把任务转成可验证目标（先写复现/验收检查再使其通过），循环到验证通过。

## 三条铁律

1. **Codex 默认只审查，绝不直接改代码或文件**（`sandbox="read-only"`）。
2. **所有修改统一由 Claude 落地**，避免两个 agent 同时写文件造成覆盖或上下文混乱。
3. **最终把关的模型 ≠ 写代码的模型** —— Codex 复审是最后一道闸，Claude 不得自我放行。

## 标准闭环

```text
Claude 拆解需求
→ Claude 写代码并自测
→ 提交 Codex 独立审查
→ Claude 按意见逐条修复
→ Codex 复审
→ 通过（否则回到「修复」）
```

## 何时调用 Codex 审查

- 完成**一个可独立验证的功能单元 / 一组相关改动后**（PR 级粒度）再送审，不要每改一行就审。
- **关键模块强制送审**：见下方「场景 C」。
- 修复后**必须复审**，直到 Codex 明确回复 ✅ PASS。
- 复审**必须复用同一 Codex 会话**（携带上一轮返回的 `SESSION_ID`），让它保留初审意见，复审更连贯。初审会话 `SESSION_ID` 必须为空（`""`）。
- **SESSION_ID 丢失时**：若上一轮返回的 `SESSION_ID` 因会话崩溃、上下文溢出或工具重启而不可用，开新会话（`SESSION_ID=""`），并在 PROMPT 中附上「上一轮审查结论摘要」作为上下文补偿。同时在「本次重点」标注"复审（会话已重建，附初审摘要）"。

## 送审前准备

调用 Codex 前，Claude 先取 `git diff` 并附在 PROMPT 中，让 Codex 精准审查变更而非自行探索文件，省 token 且更准：

```bash
git diff --no-color
```

把 diff 连同「改动文件 / 改动目的 / 本次重点」嵌入 PROMPT。详见 [codex-guide.md](codex-guide.md)。

## 审查指令与 PROMPT 模板

Codex 工具的 PROMPT 必须携带完整的审查清单。标准模板如下，根据场景在「本次重点」字段标注高风险面：

````
请 review 以下代码改动（只审不改）：

**改动文件**：[文件列表]
**改动目的**：[简要描述]
**本次重点**：[如：鉴权边界 / 并发竞态 / 输入校验 / 是否引入回归 —— 没有可省略]

**Git Diff**:
```diff
[粘贴 git diff --no-color 输出]
```

**请检查**：
1. 逻辑正确性
2. 边界条件（空值 / 越界 / 非法输入 / 溢出）
3. 安全风险（注入 / 越权 / 凭证泄漏 / 敏感信息泄露）
4. 测试缺口（缺少断言 / 未覆盖边界 / 回归缺失）
5. 可维护性（命名 / 结构 / 重复代码 / 过度抽象）

**请给出明确结论**：
- ✅ PASS：代码质量良好，可以合入
- ⚠️ OPTIMIZE：有优化建议但不阻塞合入，由 Claude 评估是否采纳
- ❌ CHANGE：必须修改，以下为具体问题清单
````

## 修复与重试策略

- Claude 依据 Codex 清单**逐条修复**，只改问题相关代码，**不顺手重构无关部分**。
- 同一改动**最多 3 轮审查**（初审 → 修复 → 复审 → 修复 → 复审 → 修复 → 第 3 次复审）。若第 3 次复审仍然 ❌ CHANGE，停下并把分歧点**抛给人工裁决**，不得无限循环。注意：3 轮中每一轮复审均遵守相同的 SESSION_ID 复用规则和完整清单要求，「第 3 次复审」在流程中有时简称「终审」但无特殊规则。
- 低于 3 轮即 PASS 则正常结束，无需继续。

## 结论处理

| Codex 结论 | 处理 |
|-----------|------|
| ✅ PASS | 通过，进入下一步（合入 / 提交 / 推送） |
| ⚠️ OPTIMIZE | 可合入；评估建议，必要时修复后复审 |
| ❌ CHANGE | 必须修复 → Claude 逐条改 → 复用会话复审 → 循环直至 PASS 或触发人工裁决 |
| **无法识别**（Codex 返回内容不含 ✅/⚠️/❌ 标记） | 将结果视为 ⚠️ OPTIMIZE 处理——人工阅读 Codex 输出，提取可操作意见后决定是否修复。若结果完全无意义（乱码、截断），按工具不可用（场景 F）处理。 |

## 常见工作流程

各场景都走同一闭环 `写码自测 → git diff → codex 审 → 修复 → 复审`，下面只列**该场景的额外动作**：

### A. 新功能 / 一组改动

- 先拆需求，列出影响文件与验收标准。
- 完成一个可独立验证的单元后再送审（新会话，`SESSION_ID=""`）。
- 验收标准写入 PROMPT 的「改动目的」中，让 Codex 按标准核验。

### B. Bug 修复

- 先定位根因，别急着改。
- 修复时**补回归测试**，确保根因确实消除。
- 送审时在「本次重点」标注"根因是否真消除、有无新边界问题、断言是否有效"。
- 复审复用初审会话的 `SESSION_ID`。

### C. 关键模块强制送审

**判断标准（原则驱动——满足任一条即强制送审）：**

| 原则 | 说明 | 跨领域举例 |
|------|------|-----------|
| **不可逆** | 错了没法撤销或回滚代价极高 | 数据库迁移（DDL）、文件删除 / 覆盖写入、生产配置变更、固件烧录、远程部署 |
| **信任边界** | 处理来自本模块之外的数据，是注入 / 投毒的入口 | 用户输入解析、网络数据反序列化、文件读取 / 导入、IPC 消息、传感器数据、外部 API 响应 |
| **静默失效** | 错了不会 crash，但结果悄悄变错，难以发现 | 数值精度丢失、权限判定逻辑、配置覆盖 / 合并、缓存键冲突、浮点比较、正则回溯 |
| **级联影响** | 错了会扩散到其他模块 / 系统 / 消费者 | 公共库 API 签名变更、共享类型 / 协议定义、事件 / 消息队列 schema、微服务间契约、插件接口 |
| **密钥与加密材料** | 密钥生成 / 存储 / 签名 / 哈希选型出错导致安全性崩塌 | 密码哈希、Token 签发、证书校验、加密算法选型、随机数生成、密钥派生 |

这些原则适用于**任何编程领域**：系统编程（FFI / 内存管理 / 信号处理）、游戏（存档完整性 / 物理确定性 / 反作弊）、数据管道（ETL 变换 / Schema 迁移 / 聚合正确性）、嵌入式（传感器输入 / 执行器控制 / OTA 固件）、CLI 工具（参数解析 / 退出码 / 管道兼容）、Web 后端（鉴权 / 支付 / 对外接口）、移动端（本地存储加密 / 生物认证 / 深度链接劫持）等。

**满足任一原则的改动 → 强制送审，即使改动很小，不得 Claude 自我放行。** PROMPT 中显式标注高风险面，必须复审到 ✅ PASS 才合入。

### D. 重构

- 前提：行为不变。先备好可对照的测试或冒烟脚本。
- 送审说明「纯重构、行为应保持不变」，请 Codex 重点核对语义漂移。
- 只改目标范围，不借机扩展功能。

### E. 不认同 Codex 的 ❌ / ⚠️

- **不盲从**：用代码 / 测试 / 规格证据反驳，在复审 PROMPT 中说明分歧及依据。
- 同一问题最多 3 轮往返，僵持则**抛人工裁决**。
- 裁决信息需包含：原始审查意见、Claude 的反驳依据、两端的分歧摘要。

### F. 工具不可用

| 错误类型 | 处理 |
|---------|------|
| `command_not_found` / `auth_required` | 提示用户 `codex login` 或安装 Codex CLI，**不重试** |
| 其他错误（`timeout` / `upstream_error` / `json_decode` 等） | 工具默认重试 1 次 |
| 仍然失败 | 若 `docs/` 或 `discuss/` 目录不存在则先创建，在其中记录 diff 与审查意图，待恢复后补审 |

## 独立决策

Codex 的意见是最后一道闸，但 Claude 仍是工程主体：

- 对明显误报需**有理有据地反驳**，而非盲从。
- 对合理意见**不得自我放行**。
- 需要第二意见时仍只用 Codex——本仓库不接入其他模型。
